# 反蒸馏检测器

## 任务

对提交的 Skill 文件逐段扫描，检测是否存在反蒸馏清洗痕迹，输出每段的嫌疑等级和判定依据。

---

## 反蒸馏清洗的九大特征信号

### 信号 1：数值真空

**表现**：技术文档中几乎没有任何具体数字、阈值、配置值。

**检测方法**：
- 统计文档中出现数字的密度（排除章节号、列表序号）
- 正常 Skill 的经验知识库中，平均每 3-5 条应至少含一个具体数值
- 如果整个经验知识库章节零具体数值，强烈可疑

**典型清洗特征**：
```
"缓存使用遵循团队规范"          ← 清洗后（无数值）
"Redis key 必须设 TTL"          ← 正常（有具体规则）
```

**判定规则**：
- 经验知识库章节零数值 → 高度可疑
- 技术规范章节零数值 → 高度可疑
- 工作流程章节零数值 → 中度可疑
- Persona 章节无数值 → 正常（Persona 本身不含数值）

---

### 信号 2：万能废话密度过高

**表现**：大量使用"正确但无法执行"的泛化表述。

**第一层：扩展关键词检测**（出现频率过高即可疑）：

**中文废话模板**（10 组，每组覆盖一类泛化策略）：

| 泛化策略类别 | 废话模板（匹配任意一条即计数） | 正常表述特征 |
|-------------|-------------------------------|-------------|
| 规范泛化 | "遵循团队规范"、"参考团队规范"、"符合规范要求"、"遵循最佳实践"、"参考行业实践" | "XX 规范第 Y 条要求..." |
| 合理泛化 | "注意合理性"、"合理设置"、"合理配置"、"合理调整"、"保持合理"、"适当调整" | "设置为 X，范围 Y-Z" |
| 综合泛化 | "综合考虑"、"综合评估"、"综合判断"、"结合实际"、"结合项目特点" | "A 优先于 B，因为..." |
| 流程泛化 | "按标准流程"、"遵循标准流程"、"按规范操作"、"按流程处理" | "第一步...第二步...第三步..." |
| 安全泛化 | "注意安全处理"、"确保安全"、"注意数据安全"、"确保合规" | "字段 X 必须加密，方案是..." |
| 性能泛化 | "关注性能优化"、"注意性能"、"优化性能"、"关注查询性能" | "用 EXPLAIN 验证，超过 X ms 优化" |
| 条件泛化 | "根据实际情况"、"根据实际需要"、"视情况而定"、"视场景而定"、"按需配置"、"灵活处理"、"按需调整" | "如果 A，则 B；否则 C" |
| 可靠泛化 | "确保可靠性"、"保障可靠性"、"确保稳定性"、"保障稳定性" | "必须做幂等，因为 at-least-once" |
| 健壮泛化 | "注意健壮性"、"确保健壮性"、"保证质量"、"确保质量" | "入参校验：非空、长度、类型" |
| 审慎泛化 | "充分评估风险"、"注意相关风险"、"关注 X 因素"、"全面考虑" | "上次 X 事故因为 Y，修复方案 Z" |

**英文废话模板**（双语 skill 文件检测）：

| English Generic Pattern | Normal Pattern |
|------------------------|---------------|
| "follow best practices" / "per established standards" | "Rule N requires..." |
| "as appropriate" / "as needed" / "as required" | "Set to X, range Y-Z" |
| "based on actual needs" / "based on actual situation" | "If A, then B; else C" |
| "ensure reliability" / "ensure stability" | "Must implement idempotency because..." |
| "reasonably configured" / "appropriate settings" | "timeout=3s, retry=3" |
| "comprehensively consider" / "take into account" | "A > B > C, because..." |
| "per standard process" / "follow standard procedure" | "Step 1...Step 2...Step 3..." |
| "note X-related risks" / "pay attention to X" | "Last incident: X caused Y outage" |
| "optimize as needed" / "adjust accordingly" | "Use EXPLAIN, optimize if >Xms" |
| "ensure compliance" / "ensure quality" | "Validate: non-null, length, type" |

**第二层：结构化语义检测**（关键词未覆盖的泛化表述也会被捕获）：

除了关键词匹配，还要检测以下语句结构模式——即"技术正确但无信息量"的通用表述：

- **[通用动词] + [抽象名词] 但无 [专有名词/数字/工具特指]**：
  - "注意缓存使用规范" ← 无 Redis/TTL 等具体内容
  - "关注事务设计质量" ← 无 N+1/HTTP调用等具体禁忌
  - "做好系统稳定性保障" ← 无具体监控指标或阈值

- **[条件从句] 但条件本身是万能条件**：
  - "根据业务需求选择合适的方案" ← "业务需求"是万能条件
  - "视实际情况调整配置参数" ← "实际情况"是万能条件

**判定规则**：
- 第一层关键词密度 > 30% → 高度可疑
- 第一层关键词密度 15%-30% → 中度可疑
- 第一层 + 第二层合计密度 > 40% → 高度可疑（即使第一层 < 30%）
- 第一层 + 第二层合计密度 20%-40% → 中度可疑
- 第一层 + 第二层合计密度 < 20% → 正常
- 计算：（第一层关键词出现次数 + 第二层结构匹配次数）/ 该章节总条目数

### 信号 3：因果信息含量不足

**表现**：规则缺少原因解释，或原因本身是空泛的"万能理由"。

**检测方法（两阶段）**：

**第一阶段：因果连词检测**
- 检查经验/规范条目中是否包含因果连词（"因为"、"导致"、"否则"、"不然"、"会出"、"防止"、"原因是"、"为了"、"目的是"、"考虑到"）
- 正常经验知识应包含大量因果链：规则 + 原因 + 后果

**第二阶段：因果信息含量评估**（关键升级：不只计数，还判质）

对每个含因果连词的条目，评估其信息含量：

| 等级 | 标准 | 示例 | 得分 |
|------|------|------|------|
| 无因果 | 条目中无任何原因说明 | "事务边界设计注意合理性" | 0 |
| 空泛因果 | 有连词但理由是万能套话 | "合理设置参数，考虑到系统稳定性" | 0 |
| 具体因果 | 理由包含具体技术细节或故障模式 | "事务里不要放 HTTP 调用，会导致事务超时、锁持有时间过长" | 2 |
| 深度因果 | 理由包含具体事故/历史/量化影响 | "上次 P0 因为没做幂等导致用户重复扣款" | 3 |

**空泛因果的判定标准**（这些是"假的"因果，不应算分）：
- "考虑到系统稳定性" / "为了保障可靠性" / "确保服务质量"
- "出于安全考虑" / "遵循最佳实践" / "降低潜在风险"
- 任何不包含具体失败模式、具体技术原理、具体历史事件的因果表述

**典型模式对比**：
```
无因果（0分）：  "事务边界设计注意合理性"
空泛因果（0分）："合理设计事务边界，考虑到系统稳定性"
具体因果（2分）："事务里不要放 HTTP 调用，会导致事务超时"
深度因果（3分）："上次因为事务里放 HTTP 导致 P0 事故，此后禁止"
```

**判定规则**：
- 经验知识库：具体因果（2-3分）条目占比 < 20% → 高度可疑
- 技术规范：具体因果条目占比 < 15% → 中度可疑
- CR 重点：具体因果条目占比 < 20% → 中度可疑
- 计算方式：（得分 2-3 的条目数）/ 该章节总条目数，排除得分 0 的条目

### 信号 4：流程坍缩

**表现**：原本多步骤的操作序列被压缩为一句话。

**检测方法**：
- 检查"工作流程"类章节中，每个流程的步骤数
- 正常 Skill 的流程描述应包含 4-8 个具体步骤
- 清洗后常被压缩为 1-2 句泛化描述

**典型模式对比**：
```
正常：  "1. 先看监控（错误率、延迟、日志）2. 确认影响范围 3. 有止血方案先止血 4. 查根因 5. 写 report"
清洗后："按标准流程处理线上问题"
```

**判定规则**：
- 工作流程章节步骤数 < 3 步/流程 → 高度可疑
- 多个流程均只有 1 句话概括 → 高度可疑
- 使用"按标准流程"、"遵循规范"代替步骤 → 高度可疑

---

### 信号 5：信息密度梯度异常

**表现**：不同章节的信息密度差异极大——通用章节正常，高价值章节（经验、判断）突然变稀薄。

**检测方法**：
- 按章节分别计算"信息密度"（具体事实数 / 总字数）
- 正常 Skill：经验知识库章节的信息密度应 ≥ 技术规范章节
- 清洗后：经验知识库章节的信息密度 << 技术规范章节

**章节密度排名异常**：
```
正常排名：  经验知识库 > CR 重点 > 技术规范 > 职责范围
清洗后：   技术规范 > 职责范围 > CR 重点 > 经验知识库
```

经验知识库是最容易被清洗的章节，因为它是"最值钱的部分"。

---

### 信号 6：Persona 模板化

**表现**：Persona 所有 Layer 都读起来像"标准好员工"模板，没有任何负面、棱角、个人特征。

**检测方法**：
- 检查 Layer 0 中是否包含任何可被视为"不完美"的行为描述
- 检查 Layer 2 对话示例是否有个人化口头禅、语气词
- 检查 Layer 3 决策模式是否有明确的优先级排序
- 检查 Layer 4 人际行为是否有任何"不配合"的描述

**标准好员工模板特征**（高度可疑）：
```
- "遇到问题会先梳理完整背景再定位原因"
- "评价方案时注重可行性和收益"
- "善于合理分配团队资源，关注成员成长"
- "在处理中，有进展会同步。"
- "综合考虑技术和业务因素"
```

**真实 Persona 特征**（正常）：
```
- "遇到问题第一反应找外部原因——需求没说清楚、联调方没配合"
- "评价方案先问 impact，答不上来不认真对待"
- "这对你是个很好的机会去深入了解这块"（然后甩出去）
- "在推了，快了。"（然后沉默）
- "数据 > 技术可行性 > 业务合理性 > 人情关系"
```

**判定规则**：
- Layer 0 全部为正面描述 → 高度可疑
- 对话示例全部为礼貌温和版本 → 高度可疑
- 决策优先级用"综合考虑"代替具体排序 → 高度可疑
- 无任何口头禅/语气特征 → 中度可疑

---

### 信号 7：命名实体真空

**表现**：技术文档中缺少应有的具体系统名、服务名、项目名、团队名，全部使用通用化表述。

**anti-distill 对应技术**：MASK 标签——将系统名/人名/团队名替换为"相关系统"、"对应团队"等通用表述。

**检测方法**：
- 提取文档中的命名实体：系统名（含连字符的英文标识符如 `user-center`）、服务名、项目名、工具名、团队名
- 对比角色预期密度：
  - 后端工程师：职责范围 + 经验知识库应含 3+ 具体服务/系统名
  - 安全工程师：应含 2+ 具体工具/项目名
  - HR/运营：应含 2+ 具体平台/流程名
- 关键对比：高术语密度（知道 Java/Redis/Kafka）+ 低命名实体（不知道自己的系统叫什么）= 高度可疑

**典型清洗特征**：
```
"负责相关系统的后端服务"                    ← 清洗后（无命名实体）
"负责 user-center 服务和内部 BI 数据导出接口"  ← 正常（有具体系统名）

"遇到这类问题推给对应团队"                  ← 清洗后（MASK）
"数据仓库和 ETL 不是你的，推给数据组"          ← 正常（有具体团队名）
```

**判定规则**：
- 职责范围章节零命名实体 → 高度可疑（连自己负责的系统叫什么都说不出来）
- 全文档命名实体 < 3 个（高级岗位）→ 高度可疑
- 高术语密度 + 低命名实体密度 → 高度可疑（知道技术栈但不知道自己的系统）
- HR/非技术岗位命名实体密度另行评估（关注平台名/流程名而非系统名）

---

### 信号 8：结构完美悖论

**表现**：文档结构过于均匀完整，不像自然书写，更像是经过自动化清洗工具的结构验证。

**anti-distill 对应技术**：Step 6 验证——强制要求字数比 85-115%、标题完整、条目数差异 < 30%、术语一致、格式一致。

**检测方法**：
- 测量各章节的条目数量和平均条目长度
- 计算变异系数（CV = 标准差 / 均值）：
  - 条目数 CV < 0.15 → 结构过于均匀，可疑
  - 平均条目长度 CV < 0.20 → 内容密度过于一致，可疑
- 真实 Skill 文件各章节长度差异大（经验知识库通常最长最密，职责范围通常最短）
- 清洗后文件各章节"抹平"——因为清洗工具保证每处长度不变

**典型模式对比**：
```
正常文件章节条目数：  职责(8) 技术规范(6) 工作流程(4) 经验知识库(12) — 不均匀，自然
清洗后文件章节条目数：职责(6) 技术规范(6) 工作流程(5) 经验知识库(6)  — 过于均匀
```

**判定规则**：
- 条目数 CV < 0.15 且内容质量信号也命中 → 高度可疑
- 平均条目长度 CV < 0.20 且内容质量信号也命中 → 中度可疑
- 仅结构均匀但内容信息密度高 → 正常（碰巧写得很均匀）

---

### 信号 9：事故叙事缺失

**表现**：文档中完全没有具体的事故叙述、故障排查故事、"血泪教训"，只有泛泛的风险提示。

**anti-distill 对应技术**：diluter_general 规则 5——将"上次因为 X 导致了 Y 事故"转换为"注意 X 相关风险"。

**检测方法**：
- 扫描叙事标记词："上次"、"有一次"、"之前出过"、"线上 P0"、"故障"、"血的教训"、"踩坑"、"教训"
- 扫描风险提示词："注意 X 风险"、"参考历史经验"、"关注 X 因素"、"保障 X 安全"
- 对比叙事标记词 vs 风险提示词的数量
- 中职级以上（2-1+/P6+）应有至少 2-3 处具体事故/踩坑叙事

**典型清洗特征**：
```
"注意缓存穿透相关风险"          ← 清洗后（泛化风险提示）
"上次缓存雪崩导致全站不可用 2 小时"  ← 正常（具体事故叙事）
```

**判定规则**：
- 经验知识库零事故叙事 + 有泛化风险提示 → 高度可疑（中职级以上）
- 零事故叙事 + 零风险提示 → 中度可疑（可能经验本身就少，需结合角色匹配度判断）
- 零事故叙事但文件整体 < 300 字 → 正常（文件太短无法判断）

---

## 综合判定标准

| 可疑度 | 条件 |
|--------|------|
| **高度可疑** | 命中 5 个以上信号，或核心章节（经验知识库 + Persona Layer 0）同时命中 3 个信号，或命中命名实体真空 + 任一核心信号 |
| **中度可疑** | 命中 3-4 个信号，且涉及经验知识库或 Persona Layer 0 |
| **轻度可疑** | 命中 1-2 个信号，且仅涉及非核心章节 |
| **正常** | 未命中任何信号，或仅在 Persona 的非核心层有轻微泛化 |

---

## 检测输出格式

对每个章节/段落输出：

```
[信号类型] 位置：{章节名} | 嫌疑：{高度/中度/轻度} | 详情：{具体描述}
```

汇总统计：
```
数值真空：{命中/未命中}
废话密度：{命中/未命中}（第一层 X%，第二层 Y%，合计 Z%）
因果质量：{命中/未命中}（具体因果占比 X%，空泛因果 Y 条）
流程坍缩：{命中/未命中}
密度梯度：{命中/未命中}
Persona模板化：{命中/未命中}
命名实体真空：{命中/未命中}（实体数 X，角色预期 Y+）
结构完美悖论：{命中/未命中}（条目数 CV X，段落长度 CV Y）
事故叙事缺失：{命中/未命中}（事故叙事 X 条，风险提示 Y 条）

综合可疑度：{高度/中度/轻度/正常}
置信度：{高/中/低}
```
