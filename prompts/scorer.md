# 信息密度评分器（v2.0 优化版）

## 任务

对 Skill 文件进行多维评分，量化其信息含量，判断是否可能被清洗过。评分结果作为检测结论的量化依据。

> **v2.0 优化要点**：(1) 术语维度从"专业度"改为"语境化"并降权，消除清洗文件送分；(2) Persona 鲜活度权重从 1.5 提升至 3.0，成为最强行为信号；(3) 角色匹配度改为自洽推断，不依赖 meta.json；(4) 新增职级/岗位分级阈值，减少误报。

---

## 评分维度（共 7 项，每项 0-10 分，权重已优化）

### 维度 1：具体事实密度（权重 3.5，原 3.0）

**衡量标准**：文档中包含多少可执行的具体事实（数字、阈值、配置、工具名、命令、步骤）。

| 分数 | 标准 |
|------|------|
| 8-10 | 经验知识库中大量具体规则，带数值、阈值、命令、工具名 |
| 5-7 | 有一些具体信息，但部分章节偏泛化 |
| 3-4 | 仅技术栈和基础规范有具体信息，经验部分全为泛化描述 |
| 0-2 | 几乎没有任何具体事实，全文泛化 |

**计数方法**：
- 统计经验知识库章节中含具体数字的条目占比
- 统计含具体工具/组件名的条目占比
- 统计含明确"必须/不要/避免"指令的条目占比
- 三个占比取平均，映射到 0-10

---

### 维度 2：因果完整性（权重 2.5，原 2.0）

**衡量标准**：不只看因果连词数量，而是评估因果链的信息含量。清洗文件可能伪造"考虑到系统稳定性"等空泛因果，需要区分空泛因果和具体因果。

| 分数 | 标准 |
|------|------|
| 8-10 | 大部分规则都有具体原因（含技术细节）和具体后果（含失败模式） |
| 5-7 | 关键规则有具体原因，部分没有 |
| 3-4 | 有因果连词但原因空泛（"考虑稳定性"/"保障可靠性"） |
| 0-2 | 全部只有"要/不要/注意"，无任何原因 |

**信息含量分级**（每条因果链评分）：
- 0 分（无因果）："事务边界设计注意合理性"
- 0 分（空泛因果）："合理设计事务边界，考虑到系统稳定性"
- 2 分（具体因果）："事务里不要放 HTTP 调用，会导致事务超时"
- 3 分（深度因果）："上次因为事务里放 HTTP 导致 P0 事故，此后禁止"

**计数方法**：
- 计算得分 2-3 的条目占比（排除得分 0 的空泛因果）
- ≥ 50% → 8-10 分
- 30%-50% → 5-7 分
- 10%-30% → 3-4 分
- < 10% → 0-2 分

---

### 维度 3：流程颗粒度（权重 2.0，不变）

**衡量标准**：工作流程描述的细化程度。

| 分数 | 标准 |
|------|------|
| 8-10 | 每个流程 5+ 步骤，带具体操作、检查点、判断条件 |
| 5-7 | 流程 3-4 步，有操作但缺乏判断条件 |
| 3-4 | 流程 2-3 步，操作模糊 |
| 0-2 | 流程被压缩为 1 句话或"按标准流程" |

**计数方法**：
- 统计工作流程章节中每个流程的步骤数
- 平均步骤数 ≥ 5 → 8-10 分
- 平均步骤数 3-4 → 5-7 分
- 平均步骤数 2 → 3-4 分
- 平均步骤数 < 2 → 0-2 分

---

### 维度 4：Persona 鲜活度（权重 3.0，原 1.5 — 大幅提升）

**衡量标准**：Persona 是否有辨识度、有棱角、有个人特征，还是"标准好员工"模板。

**重要说明**：colleague-skill 的 intake→persona_builder 流程会自动生成高辨识度 Persona（通过标签翻译表和行为规则模板）。因此 bland Persona 在 colleague-skill 格式中几乎必然是被清洗过的。此权重从 1.5 大幅提升至 3.0。

| 分数 | 标准 |
|------|------|
| 8-10 | 有明确个人特质（正负面均可）、独特口头禅、具体对话示例、个性优先级排序 |
| 5-7 | 有一些个人特征，但不够鲜明 |
| 3-4 | 基本都是正面描述，仅有少量个人色彩 |
| 0-2 | 完全是"标准好员工"模板，任何人都能写出来 |

**检查清单**（每项 +1 分，满分 10）：
1. Layer 0 包含可被视为"不完美"的行为描述（如甩锅、拖延、反问）
2. 有高辨识度的口头禅或常用语（超出"对齐/拉通"等通用企业黑话）
3. 有具体的对话示例（含语气/措辞/沉默等细节）
4. 决策模式有明确的优先级排序（如 X > Y > Z）
5. 人际行为有"不配合"或"有策略"的描述
6. 包含压力下的行为变化描述
7. 有明确的"雷区"或"触发条件"
8. 表达风格有独特的句子长度/语气特征
9. 对不同对象有不同态度（上级/平级/下属）
10. **独特性测试**：是否有任何同角色随机人物无法生成的信息（如 tianyi 的杀戮尖塔术语、jiaxiu 的星座分析框架）

---

### 维度 5：上下文丰富度（权重 2.0，不变）

**衡量标准**：是否包含历史背景、架构决策原因、隐性知识等"只可意会"的内容。

| 分数 | 标准 |
|------|------|
| 8-10 | 大量历史背景、决策原因、"看起来 X 但实际 Y"的隐性知识 |
| 5-7 | 有一些背景说明和隐性知识 |
| 3-4 | 偶尔提及背景，大部分内容是表面规则 |
| 0-2 | 纯表面规则，没有任何"为什么是这样"的解释 |

**检查关键词**：
- "当初"、"历史原因"、"之前"、"之前出过"、"看起来...但"、"实际上"、"虽然...但"、"不要动...因为"、"当时"
- ≥ 5 处 → 8-10 分
- 2-4 处 → 5-7 分
- 1 处 → 3-4 分
- 0 处 → 0-2 分

---

### 维度 6：角色自洽匹配度（权重 1.0，原 1.5 — 降权并重设计）

**衡量标准**：信息深度是否与文档自身展现出的资历水平自洽。不依赖 meta.json（可被伪造），而是从文档内容推断资历。

| 分数 | 标准 |
|------|------|
| 8-10 | 经验深度明显超出书本，有大量实战沉淀 |
| 5-7 | 经验深度与角色匹配，有实战痕迹 |
| 3-4 | 经验停留在通用层面，感觉像"刚整理的笔记"而非"多年积累" |
| 0-2 | 完全是教科书/文档搬运级别，不像有实际工作经验的人写的 |

**自洽评估逻辑**：
- 如果文档提到负责多个系统/模块 → 推断为高级 → 应有大量隐性知识
- 如果文档提到负责单一模块 → 推断为中级 → 应有明确的踩坑经验
- 如果文档仅列技术栈无系统描述 → 推断为初级 → 经验可能确实稀薄
- 如果文档展现的资历（系统复杂度、决策深度）与信息密度不匹配 → 扣分

---

### 维度 7：术语语境化（权重 0.5，原"术语专业度"权重 1.0 — 降权重定义）

**衡量标准**：不只看术语是否出现，而是看术语是否在具体语境中使用。anti-distill 明确保留术语列表（classifier.md "safe zone"），所以单纯出现术语名称不能说明文件未被清洗。必须看术语是否附带具体使用规则、配置细节或操作命令。

| 分数 | 标准 |
|------|------|
| 8-10 | 术语出现在具体使用场景中："Redis key 必须设 TTL"、"Kafka 消费者必须做幂等" |
| 5-7 | 术语出现在技术栈列表中，部分在经验规则中有具体使用说明 |
| 3-4 | 术语仅出现在技术栈列表中，经验部分用"缓存"/"消息队列"替代 |
| 0-2 | 全文用词泛化，几乎看不到具体技术术语 |

**关键区分**：
- 高分："Redis key 必须设 TTL" — 术语 + 具体规则
- 低分："缓存使用遵循团队规范" — 术语被泛化词替代
- 零分陷阱：技术栈写了"Redis、Kafka"但经验部分全用"缓存"/"消息队列"替代 → 实际得分 3-4 而非 8-10

## 职级/岗位分级阈值（误报缓解）

不同职级和岗位的信息密度天然不同，需调整判定标准以减少误报。

### 按职级分级

| 职级 | 预期事实密度 | 预期 Persona 鲜活度 | 检测灵敏度 |
|------|------------|-------------------|-----------|
| 初级（P4-P5/1-1~2-1） | 低：4-6 分即可 | 低："性格还在形成" | 放宽，需多 1 个额外信号才标记可疑 |
| 中级（P6/2-2~3-1） | 中：5-7 分预期 | 中：应有部分可见特质 | 标准阈值 |
| 高级（P7+/3-2+） | 高：7-10 分预期 | 高：必须有鲜明个性 | 收紧，任何维度 < 6 分即可疑 |

### 按岗位领域分级

| 岗位 | 预期数字密度 | 预期命名实体 | 特殊考量 |
|------|-----------|-----------|---------|
| 后端工程师 | 高（配置值、阈值、步骤数） | 高（服务名、系统名） | 标准检测 |
| 安全工程师 | 中高（监控指标、误伤率） | 高（工具名、项目名） | 关注安全事故叙事 |
| HR/运营 | 低（日期、人数等） | 中（平台名、流程名） | 数字密度阈值放宽，关注流程细节 |
| 产品/设计 | 低（无配置值） | 中（产品名、项目名） | 关注决策逻辑和用户洞察 |
| 管理/PMO | 中（里程碑、预算） | 中（团队名、项目名） | 关注人际网络和上下文 |

---

## 短文件分级处理

| 字数范围 | 处理方式 | 置信度 |
|---------|---------|--------|
| < 200 字 | 不可评估，要求补充更多内容 | 无 |
| 200-500 字 | 可评估但置信度降低，在报告中标注"低置信度" | 低 |
| 500+ 字 | 标准评估 | 标准 |

## 综合评分

### 加权总分计算

```
总分 = (具体事实密度 × 3.5 + 因果完整性 × 2.5 + 流程颗粒度 × 2
      + Persona鲜活度 × 3.0 + 上下文丰富度 × 2 + 角色匹配度 × 1.0
      + 术语语境化 × 0.5) / 14.5
```

权重说明：
- **Persona 鲜活度权重从 1.5 提升到 3.0**：colleague-skill 的 intake→persona_builder 流程必然生成高辨识度 Persona， bland Persona 几乎必然被清洗
- **术语从"专业度"改为"语境化"且权重从 1.0 降至 0.5**：anti-distill 明确保留术语，此维度对检测无贡献，反而膨胀清洗文件分数
- **角色匹配度从 1.5 降至 1.0**：依赖 meta.json 可被伪造

### 总分对应判定

| 总分 | 判定 | 说明 |
|------|------|------|
| 6.5+ | 正常 | 信息丰富，未发现清洗痕迹 |
| 4.5-6.4 | 轻度可疑 | 部分章节偏泛化，可能是写作风格 |
| 2.5-4.4 | 中度可疑 | 多个维度明显不足，存在清洗痕迹 |
| < 2.5 | 高度可疑 | 信息严重不足，极大概率被清洗 |
